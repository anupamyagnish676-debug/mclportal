1. Executive Introduction

The Mahanadi Coalfields Limited (MCL) Internship Portal is an enterprise web application designed to digitalize, streamline, and secure the end-to-end internship workflow. It replaces paper-based Letter of Recommendation (LOR) submission, manual attendance ledger maintenance, physical logbook filing, and analog certificate stamping.

For MCL’s Human Resource Development (HRD) department and Area Training Offices, the portal provides complete visibility into student placements, mentor allocations, progress tracking, and certificate issuance. It ensures zero forgery, automates notifications, and presents a modern visual experience aligned with the corporate identity of MCL.

💡 Key Business Benefit: Reduces the administrative overhead of managing an intern from LOR approval to joining and final evaluation by over 85%, while completely securing certificates from tampering via real-time database-backed QR verification.

2. System Architecture & Tech Stack

The portal is constructed utilizing a highly optimized modern web stack, dividing processing workload between client-side rendering and secure serverless backend environments.

Tech Stack Overview

Component Technology Purpose
Framework Next.js 14 (React) Handles both Client-Side routing (SPA experience) and Server-Side execution (Secure APIs and Middleware).
Database Supabase (PostgreSQL) Stores transactional data, RLS security policies, and manages user accounts.
Object Storage Supabase Storage Host study materials, student project reports, LOR files, and issued certificates.
Email Service NodeMailer + Google SMTP Sends dynamic credentials, joining letters, and certificate attachments.
PDF Engine pdf-lib Draws certificates and logos dynamically on the server side to protect raw signatures.
Styling Vanilla CSS + Tailwind Sleek glassmorphism effects, cinematic Ken Burns grids, and modern layout animations.

Architectural Security Boundary: The frontend client speaks to Supabase through Server-Side API Routes. This ensures that secret API credentials (such as the Supabase Service Role Key) are never exposed to the student or employee's browser, preventing database manipulation.

3. Database Schema & Relationships

The relational database structure ensures strict data integrity. The schema utilizes cascade rules on delete actions to maintain cleanup safety.

Table Schemas

1. Table: profiles

Stores identity data for all user roles (Students, Mentors, Employees, Admins).

Column Name Data Type Constraints Description
id UUID Primary Key (References auth.users) Unique User Authentication ID.
full_name Text Not Null Full name of the user.
email Text Not Null, Unique Login email address.
role Text Not Null Enum: student, mentor, employee, admin.
area Text Nullable Specific operational area placement (e.g. Talcher, HQ).
signature_data Text Nullable Base64 encoded string representing the hand-drawn signature.

2. Table: applications

Contains Letter of Recommendation (LOR) submissions forwarded by Employees.

Column Name Data Type Constraints Description
id UUID Primary Key Unique application ID.
status Text Not Null pending_hq, pending_area, approved, rejected.
student_name Text Not Null Name entered during LOR upload.
student_email Text Not Null Student's email address.
roll_no Text Nullable Roll number/Reg number.
university Text Nullable University/College name.
student_id UUID References profiles(id) ON DELETE SET NULL Links application to student account once registered.

3. Table: internships

Defines active student training periods, guidance mentors, and serial numbers.

Column Name Data Type Constraints Description
id UUID Primary Key Unique internship record identifier.
student_id UUID References profiles(id) ON DELETE CASCADE Linked student user.
mentor_id UUID References profiles(id) ON DELETE SET NULL Assigned guiding mentor.
serial_no Text Unique, Not Null Auto-incrementing serial string used for certificate registry.
start_date Date Not Null Joining date.
end_date Date Not Null Completion date.
certificate_url Text Nullable Signed link to issued certificate PDF.

4. Child Tables (Cascade Deletion Mapping)

  • attendance: Stores daily attendance records. Rows are keyed with internship_id referencing internships(id) ON DELETE CASCADE.
  • leaves: Manages leave submissions. Columns include internship_id referencing internships(id) ON DELETE CASCADE.
  • logbooks: Contains daily learning diary entries. Column internship_id references internships(id) ON DELETE CASCADE.
  • submissions: Handles files submitted for assignments. References `student_id` and `assignment_id` with cascading constraints.

4. System Security & Verification

The security implementation is divided into request validation and programmatic data protection to comply with enterprise IT standards:

A. Role-Based Access Control (RBAC)

All client routing is gated by a next-gen server-side middleware (middleware.ts). Every HTTP page request fetches the user's role:

  • Requests targeting /admin/* require role = 'admin'.
  • Requests targeting /mentor/* require role = 'mentor'.
  • Requests targeting /student/* require role = 'student'.
  • Requests targeting /employee/* require role = 'employee'.

If authorization checks fail, the middleware immediately cancels the route request and redirects the user to the /login view.

B. Supabase Row Level Security (RLS)

PostgreSQL tables have RLS policies enabled. The database evaluates policies at the row level: 

-- Example RLS Policy for Leaves table
CREATE POLICY "Students can view only their own leaves"
ON public.leaves FOR SELECT
USING (
  internship_id IN (
    SELECT id FROM public.internships WHERE student_id = auth.uid()
  )
);

This configuration guarantees that even if a student intercepts raw database requests, they cannot read, write, or alter entries belonging to other students.

C. Tamper-Proof Certificate Verification

Certificates generated by the portal are secured by a verification loop:

  1. During generation, the server creates a unique UUIDv4 for the certificate.
  2. The server generates an offline QR code containing a secure URL pointing back to the verification route: https://[portal-domain]/verify/[internship-id].
  3. This QR code is drawn directly in the center-bottom footer of the certificate PDF.
  4. When scanned, the verification route reads the UUID directly from the PostgreSQL database and presents the student's authentic training metadata (Name, Project Title, and Internship Period). Any unauthorized manual modifications made to the PDF document will not match the official database record.

5. Role-Based Portals & Workflows

The portal separates privileges into five custom roles, each possessing targeted tools:

Employee
Can submit LOR files along with details of the referred student (Name, Email, Roll Number, College). Employee sees a dashboard showing the screening progress of their referred candidates.

Global Admin (Headquarters)
Possesses system-wide visibility. Handles initial LOR screening, and routes eligible LOR requests to the concerned local operational Areas. Manages department configurations, performs bulk onboarding, and runs settings updates.

Area Admin (Local Area)
Manages local area approvals. Approves incoming routed applications. Upon approval, registers the student account inline, which automatically creates Supabase Auth credentials, calculates the serial number, and dispatches a single unified Joining Letter.

Project Mentor
Oversees assigned interns. Configured with a dedicated settings page containing an interactive signature pad to store their sign-off image. Mentors verify daily logbooks, review assignment files, mark attendance, and approve final certificate generation.

Student Intern
Views placement details, logs work daily in their logbook, submits leaves, uploads assignment files, reads study materials uploaded by their mentor, and downloads their final certificate once issued.

6. Installation & Deployment

Follow this process to configure the environment and run or deploy the portal.

A. Environment Variables (.env.local)

Create a configuration file in the project root containing the following variables: 

# Supabase API Keys (Server and Client access)
NEXT_PUBLIC_SUPABASE_URL=https://[your-project-id].supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbGciOiJIUz...
SUPABASE_SERVICE_ROLE_KEY=eyJhbGciOiJIUz...

# Email configuration
GMAIL_USER=mclinternshipportal@gmail.com
GMAIL_PASS=lshjomvfmyvtednd

B. Local Development Commands

# 1. Install required packages
npm install

# 2. Run local Next.js server in development mode
npm run dev -- -p 3001

# 3. Create production-ready compiled bundle
npm run build

# 4. Start production server locally
npm start

C. Vercel Deployment

  1. Connect your GitHub repository to your Vercel Dashboard.
  2. Add the environment variables listed above in the Environment Variables tab under Project Settings.
  3. Click Deploy. Vercel will build, optimize, and serve the site globally over HTTPS.

7. Administration & Maintenance

Operational instructions for portal admins to perform maintenance tasks:

1. Updating the Area Training Officer / GM Signature

The certificate requires three signature lines: **Project Mentor** (dynamic), **Area Training Officer** (dynamic), and **General Manager (HRD)** (static default).

  • To change the static default **GM HRD signature**, replace the image located at /public/gm-signature.png with a new transparent PNG file keeping the aspect ratio around 2.90.
  • To change the **Area Training Officer signature**, log in as the Area Admin, go to **Settings**, sign in the digital drawing pad canvas, and click **Save Signature**.

2. Manual Student Account Deletion

If you need to delete a student account directly inside the Supabase Console, the database will handle related child tables automatically:

  • Deleting a profile row in profiles will cascade and delete the corresponding row in internships, which in turn cascades and deletes all related rows in attendance, leaves, logbooks, and submissions.
  • The linked row in the applications table will not be deleted, but its student_id column will automatically be set to NULL, preserving the historical LOR submission.
