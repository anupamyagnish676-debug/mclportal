1. Executive Summary

The Mahanadi Coalfields Limited (MCL) Internship Management Portal is a high-performance web application engineered to centralize, automate, and secure the end-to-end administration of student interns across all operational Areas and Headquarters of MCL.

Historically, processing LOR referrals, managing student files, collecting attendance logs, and issuing training certificates involved heavy manual work across several offices, creating data delays and security concerns. The portal resolves these inefficiencies by establishing a secure digital platform accessible by Employees, local Area Training Officers, Mentors, and Students.

🎯 Key Business Deliverable: An administrative workspace that automates credential provisioning, attendance range validation, logbook oversight, and prints certificates with transparent logos, digital signatures, and database-matched validation QR codes.

3. Database Schema Specifications

The relational database schema is configured on **Supabase (PostgreSQL)**. Key constraints and foreign keys are set up with `ON DELETE CASCADE` or `ON DELETE SET NULL` behaviors to ensure data integrity during deletions.

1. Core Table: profiles

Stores identity and system privileges for all user categories.

Column Name Data Type Constraints Description
id UUID Primary Key (References auth.users) Unique User Auth ID.
full_name Text Not Null User's first and last name.
email Text Not Null, Unique Registered corporate or student email address.
role Text Not Null Privilege level: student, mentor, employee, admin.
area Text Nullable Assigned operational location (e.g. Talcher, Ib Valley).
signature_data Text Nullable Base64 PNG string representing the hand-drawn signature.

2. Core Table: applications

Maintains Letter of Recommendation (LOR) registrations submitted by employees.

Column Name Data Type Constraints Description
id UUID Primary Key Unique application ID.
status Text Not Null Status enum: pending_hq, pending_area, approved, rejected.
student_name Text Not Null Student's full name.
student_email Text Not Null Contact email address.
roll_no Text Nullable Student's university roll number.
university Text Nullable Attending college or university.
student_id UUID References profiles(id) ON DELETE SET NULL Links the LOR to the registered student user account.

3. Core Table: internships

Defines active student training configurations, guidance mentors, and serial numbers.

Column Name Data Type Constraints Description
id UUID Primary Key Unique internship record identifier.
student_id UUID References profiles(id) ON DELETE CASCADE Linked student user.
mentor_id UUID References profiles(id) ON DELETE SET NULL Assigned guiding mentor.
serial_no Text Unique, Not Null Auto-incrementing serial string used for certificate registry.
start_date Date Not Null Joining date.
end_date Date Not Null Completion date.
certificate_url Text Nullable Signed link to issued certificate PDF.

4. Security Implementation & Audits

The portal is built around a secure architecture designed to protect sensitive data and prevent unauthorized operations.

A. Row-Level Security (RLS) Policies

RLS is enabled on all tables in PostgreSQL. When a client performs a select, insert, or update query, the database engine verifies the user's role and session ID.

  • Profiles: Authenticated users can read profiles, but users can only edit their own profile details.
  • Attendance/Logbooks/Leaves: Students can only view and manage records tied to their own `internship_id`. Mentors can view and edit records for internships where `mentor_id = auth.uid()`.
  • Administrators: Global and Area Admins have a wildcard override policy allowing full read/write access.

B. Next.js Routing Middleware (RBAC)

A centralized middleware (middleware.ts) runs on every route request. It intercepts the user's session JWT token, decodes their role, and verifies they have permission to access the requested page: 

// middleware.ts routing rules
if (pathname.startsWith('/admin') && profile.role !== 'admin') {
  return NextResponse.redirect(new URL('/login', request.url))
}

This prevents URL manipulation attacks where a student might try to access admin dashboards by changing the URL parameters.

C. Cryptographic Signature Safeguard

Signatures for Mentors and Area Training Officers are drawn via canvas vectors and saved in the database as base64 strings in RLS-protected tables. The system does not store signatures in a public file folder. They are drawn directly from the database memory into the PDF during generation and are never exposed as independent URLs, preventing students from accessing or copying the raw signatures.

6. Administration & Maintenance Guide

This guide outlines core operational tasks for portal administrators to keep the system running smoothly.

A. Deleting Users & Deletion Order

If you need to delete a student account directly inside the Supabase Console, the database will handle related child tables automatically:

  • Deleting a profile row in profiles will cascade and delete the corresponding row in internships, which in turn cascades and deletes all related rows in attendance, leaves, logbooks, and submissions.
  • The linked row in the applications table will not be deleted, but its student_id column will automatically be set to NULL, preserving the historical LOR submission.

B. Resetting Database for a Clean Demo

If you need to wipe out all testing data (students, mentors, LOR applications, attendance logs, logbooks, etc.) except the primary Headquarters Admin account (`admin@mcl.com`) before showing the portal to officials, run the following SQL command in your **Supabase SQL Editor**: 

-- 1. Wipe out all transactional data
TRUNCATE TABLE public.attendance RESTART IDENTITY CASCADE;
TRUNCATE TABLE public.leaves RESTART IDENTITY CASCADE;
TRUNCATE TABLE public.logbooks RESTART IDENTITY CASCADE;
TRUNCATE TABLE public.submissions RESTART IDENTITY CASCADE;
TRUNCATE TABLE public.materials RESTART IDENTITY CASCADE;
TRUNCATE TABLE public.assignments RESTART IDENTITY CASCADE;
TRUNCATE TABLE public.internships RESTART IDENTITY CASCADE;
TRUNCATE TABLE public.applications RESTART IDENTITY CASCADE;

-- 2. Delete all user profiles except the HQ Admin
DELETE FROM public.profiles WHERE email != 'admin@mcl.com';

-- 3. Delete all auth credentials except the HQ Admin
DELETE FROM auth.users WHERE email != 'admin@mcl.com';
